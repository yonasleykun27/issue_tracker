📌 Definition

What is an Issue Tracker?

The foundation — understanding the core concept

🐛
Issue Tracking
An issue tracker is a software tool that records, categorizes, assigns, and monitors problems (issues) reported within an organization — from IT bugs to operational incidents.
🎯
Core Purpose
Ensure no reported problem is lost, forgotten, or unresolved. Every issue has an owner, a priority, and a traceable lifecycle from OPEN → IN PROGRESS → RESOLVED.
🔄
The Lifecycle
OPENNewly reported incident
IN PROGRESSBeing investigated by an agent
RESOLVEDFix confirmed & closed
REJECTEDInvalid or duplicate, closed
🏢
In Our Context
Ethio Telecom employees report IT & operational problems. Admins triage and assign to support agents. Agents investigate and resolve. The entire process is tracked digitally.
⚠️ Problem Statement

Why We Built This

Challenges faced without a centralized issue tracking system

📩
Lost Reports
Issues reported via email or verbal communication were easily forgotten, lost, or duplicated with no central record.
🤷
No Accountability
It was unclear who was responsible for resolving a given problem. No assignment, no deadline, no owner.
⏳
Slow Resolution
Without priority levels, critical issues sat unattended while minor ones got immediate attention — inefficient.
📊
No Visibility
Management had zero visibility into open incidents, resolution rates, or agent workloads. No data to act on.
💡

Solution: A dedicated web platform where employees report issues, admins triage and assign via round-robin, and agents resolve — with full audit trails, analytics, and real-time notifications.

✨ Features

Key Features

Core capabilities that power the system

📝 Incident Reporting
Employees submit reports with title, description, priority level, phone, address, and optional screenshot attachment. All fields validated before submission.
🔄 Round-Robin Assignment
When a report is submitted, the system automatically assigns it to the next admin in rotation. Admins then forward to agents using the same round-robin logic.
⏱️ Pending Approval Gate
New registrations require admin approval before users can log in or submit reports. Admins see pending registrations in a dedicated queue and approve/reject each one.
⚠️ Warning System
Admins can issue formal warnings with written reasons. Each warning is logged to history. Users see a real-time bell icon in the navbar showing total warnings and all reasons.
🏖️ Leave Management
Any user (employee, agent, or admin) can toggle leave status. While on leave, they are excluded from auto-assignment rotations and marked ON_LEAVE in the system.
📊 Analytics & Export
Admin dashboard features a 14-day bar chart of open/in-progress/resolved tickets. All incidents can be exported as CSV or PDF for reporting.
🤖 Automation

Smart Automation

Features that reduce manual work and ensure fairness

🔄
Round-Robin Assignment
When a report is submitted, the system finds which admin was last assigned, then assigns to the next active admin in sequence. Ensures equal workload distribution.
🔔
Real-Time Warning Alerts
Users receive instant bell notifications in the navbar when an admin issues a warning. The dropdown lists every warning with date and reason.
📋
Audit Logs
Every action on a ticket (creation, assignment, status change) creates an IssueLog record — a full tamper-proof audit trail visible in the ticket detail page.
🔒
Delete Protection
Once a ticket is assigned to an active support agent, deletion is forbidden at both the API level (returns 400) and UI level (button hidden).

🗄️ Database

Database Design

PostgreSQL schema powering the entire application

// User model — all system actors
model User {
  id Int @id
  name String
  email String @unique
  role Role // ADMIN|AGENT|USER
  status UserStatus // ACTIVE|PENDING|...
  warningCount Int @default(0)
  warningLogs WarningHistory[]
}

// Issue model — incident reports
model Issue {
  id Int @id
  title String
  status Status // OPEN|IN_PROGRESS|...
  priority Priority // LOW|MEDIUM|HIGH
  reportedById Int // FK → User
  assignedToId Int? // FK → User (nullable)
  logs IssueLog[]
}
🔗 Key Relationships
User → Issue (reported)
User → Issue (assigned)
Issue → IssueLog (audit)
User → WarningHistory
✅ Data Integrity
✓ Cascade delete on IssueLog
✓ Cascade delete on WarningHistory
✓ Unique constraint on email
✓ Enum validation for roles & statuses
✓ Nullable FK for optional assignment
✓ Auto-timestamp on createdAt/updatedAt
🔀 Workflow

End-to-End Workflow

From registration to incident resolution — the full journey

1
Employee Registers
New user signs up with name, email, password, and role. Account is created with PENDING status and blocked from accessing the system.
2
Admin Approves Registration
Admin sees the pending request in the Approvals tab. Clicks Approve → status becomes ACTIVE. User can now log in and access the platform.
3
Employee Reports an Issue
Employee submits a report with title, description, phone, address, priority, and optional screenshot. System auto-assigns via round-robin to an active admin.
4
Admin Reviews & Forwards to Agent
Admin sees the ticket in their Incident Queue (shown in blue as auto-assigned). Opens the ticket detail, selects "Assign to Support Agent" — round-robin picks next agent.
5
Agent Resolves the Issue
Agent sees the ticket in their dashboard, investigates, updates status to IN_PROGRESS, and finally marks as RESOLVED. All actions are logged to the audit trail.
